# Standards

The app is named **Standards** (renamed from "Tunes", July 2026 — the repo folder and Supabase project id still say `tunes`). "Standards" as in the repertoire sense: the songs a musician keeps.

A personal tool for consolidating scattered musical repertoire — song lists and lead sheets currently kept on paper and scattered across digital tools, plus liked recordings currently tracked as one YouTube Music playlist per song. Those playlists work for focused/analytical listening (transcription, comparison) but get in the way of casual listening, so this app exists to hold that analytical, per-song recording list on its own, separate from casual playlists. Built for personal use first, with multi-user use as a goal.

## Language

**Song**:
A canonical, shared piece of music — title, composer, year (the year it was *written*, see [ADR-0007](adr/0007-original-dates-and-albums.md)), lyricist, etc. — genre-neutral (not jazz-specific "tune" or lyrics-specific "song" in the strict sense; covers instrumental pieces too). Any User can create a new Song when search/autocomplete finds no match, so occasional duplicates are expected and get merged by a Site Admin later rather than gated upfront. A new Song begins non-discoverable: it is available to its creator but appears in another authenticated User's Add Song search only after a Site Admin enables **Visible to all users**. Discoverability is not anonymous publication and never exposes a User's private Song data. This reverses an earlier assumption that Song data would stay fully siloed per User — see [ADR-0003](adr/0003-song-canonical-user-song-personal.md).
_Avoid_: Tune (legacy/jazz-specific term being phased out)

**song_user_data**:
A User's private layer over a canonical Song: their own notes, their own display title (can diverge from the Song's canonical title — e.g. "I Found a New Baby" instead of the canonical "I've Found a New Baby"), when they added it, and the fact that the Song is in their list. Not a *kind* of Song — it is User-specific data *attached to* a Song, not a Song itself, which is why it is named for data rather than as a "User Song" entity. One undifferentiated list regardless of mastery — gig-ready repertoire, Songs being studied/transcribed, and someday-Songs all live together; no mastery-level silos. Wherever the UI presents a Song to that User, use their display title when present and fall back to the shared title; search within their own list matches both, while cross-User Add Song discovery matches only the shared title. In the UI this is simply presented as "Songs" (the User's list) — `song_user_data` is database/glossary vocabulary, not a UI label. There is at most one private row per User and Song, and owner-scoped RLS keeps its payload private.
_Avoid_: User Song, My Tune, Listed Song, Repertoire Entry (earlier names considered and rejected — "User Song" in particular wrongly implies a subtype of Song)

**Artist**:
A canonical, shared identity that can receive a musical credit: a person, group, orchestra, choir, character, or another credited entity. This follows MusicBrainz's broad and useful meaning rather than treating Artist as synonymous with individual performer. Artist kind may be absent when the source does not know it; missing is not the same as MusicBrainz's positive `Other` type and must not be coerced to it. Composer, lyricist, writer, and performer are roles relating an Artist to a Song or Recording; they are not different entity types. A credited group remains the group unless individual members are separately known — never omit it because it is not a person, and never infer its members. The compatibility rollout has established shared `artists`, private `artist_user_data`, Artist-backed Song credits, and `recording_artist_credits`; the legacy `people` / `song_writers` storage remains temporarily behind the Song-credit compatibility surface until contraction. See [ADR-0008](adr/0008-provider-neutral-music-entities-and-user-data.md) and [direction/artist-browsing.md](direction/artist-browsing.md).
_Avoid_: using Person as the universal identity; using performer as a synonym for Artist

**artist_user_data**:
A User's private layer over a canonical Artist: personal notes, tags, and any later private organization. There is at most one row per User and Artist. It is not an Artist subtype and must not hold the shared name, kind, biography, or provider identity. `artist_user_data` follows the `song_user_data` private-layer pattern; those names are database vocabulary, not UI labels. Owner-scoped RLS keeps this payload private while the Artist identity and credit facts are shared.

**Song Credit**:
A relationship from a Song to an Artist with a role such as `composer`, `lyricist`, or generic `writer`, plus credited-as text and ordering where needed. One Artist may hold more than one role on the same Song; preserve those roles rather than flattening the credit to a list of names. “Writer” is a role, not an entity that is necessarily represented locally as a person. Structured performer credits use the same Artist identity on Recording relationships.

**Recording**:
A canonical, provider-neutral identity for one specific recorded performance or version of a Song. Within the current product model it is Song-scoped: every Recording belongs to one Song. An external catalog Recording or a long provider item can involve multiple Works/Songs; until a richer multi-Work model is deliberately designed, that source may support separate Song-scoped Recordings rather than collapsing them into one cross-Song row. A Recording can optionally match a MusicBrainz Recording, but it still exists when no catalog knows about it — for example, an informal concert video. One Recording can have several provider items and Platform Links rather than being either a YouTube Music track *or* a YouTube video. Its `name` is free text because a Recording's useful title is often a live/session descriptor (e.g. "Live at Village Vanguard, 1961") rather than just the Song title. Canonical facts include its credited-artist snapshot, performance date, duration, and structured Artist credits when known. `Recording.artist` is specifically the single editable credited-as text used for display, not an Artist foreign key: the published credit can be a group and can differ from notable individual performers. Every Recording has a Kind — see below. Canonical Recordings are readable to authenticated Users. A User who saved a Recording may edit it under the temporary last-write-wins policy. Client-side canonical deletion is not permitted.
_Avoid_: defining a Recording as a provider result; using performer as a substitute for its credited artist

**user_recording_data**:
A User's private relationship to a canonical Recording: that they saved it, their notes and rating, their preferred order within the Recording's Song, their preferred playing key and practice tempo, and other genuinely personal organization such as tags. Key and tempo here are the User's working values, not claims about the key or measured tempo of the canonical performance. There is at most one such relationship per User and Recording, enforced by the composite `(user_id, recording_id)` key; its presence means saved and its absence means unsaved. It is not a kind or copy of Recording. The UI simply shows these as the User's Recordings; `user_recording_data` is database/glossary vocabulary. Owner-scoped RLS keeps this payload private, and unsaving removes only this relationship rather than the shared Recording or provider evidence.
_Avoid_: User Recording (wrongly implies a subtype or duplicate Recording)

**Recording Kind**:
Whether a Recording is `released` (an officially released audio track or music video — eligible for cross-platform matching via Platform Links) or `video_capture` (performance footage, a broadcast rip, or another unofficial video with no commercial release to match across platforms). Deliberately not called "format": format would suggest audio versus video or the original release medium (vinyl, tape, CD, lacquer), neither of which Kind models. Also deliberately not called "type": "type" is MusicBrainz's word for classifying catalogued entities, while Kind is Standards' own release-status classification and is independent of whether MusicBrainz happens to catalogue the Recording. MusicBrainz's `video` boolean is orthogonal: both an official music video and an unofficial capture are videos, but their Standards Kinds differ. A YouTube provider-item category of `video` therefore cannot decide Recording Kind by itself; it needs an overrideable classification. A future self-hosted upload is a provider/media source, not automatically a `user_upload` Recording Kind; the release-status term for an unreleased audio-only/home recording remains to be decided if that feature is built.
_Avoid_: Format, type

**Platform Link**:
A link from a `released` Recording to where it can be played on one specific platform (Spotify, Apple Music, Amazon Music, YouTube Music, or YouTube) — one Recording can have several, at most one selected link per platform. It may come directly from the provider item through which the Recording was found or be expanded through Odesli; it is stored rather than re-queried live. Cross-platform matching is only attempted for `released` Recordings. A `video_capture` remains playable through its YouTube Item but does not pretend to have matched commercial Platform Links. Absence of a Platform Link for some platform is expected and normal, not a failure state.
_Avoid_: Streaming link, external link (too generic — Platform Link is specifically one of the matched streaming platforms, not any arbitrary URL)

**YouTube Item**:
A normalized snapshot of one selected YouTube or YouTube Music result in `youtube_items`, globally identified by the shared YouTube video ID and attached to Recording through `recording_youtube_items`. A Recording can have several YouTube Items, and one long item may support separate Song-scoped Recordings. The snapshot retains accumulated discovery provenance from YouTube Music search, official YouTube search, manual URL entry, and legacy URL migration. Its normalized search category is `song` when catalog-song/Topic evidence exists and `video` otherwise; a manual URL therefore begins conservatively as `video` and can be enriched to `song` later. YouTube Music-specific artist and album IDs/names remain nullable when that source has supplied no metadata. That provider-item category describes the available source evidence, not Recording Kind: a video may be either an official `released` music video or a `video_capture`. It is provider data supporting a Recording, not the Recording itself. Shared snapshots and associations are client-readable but can be merged or created only through the bounded authenticated save operation; ordinary clients cannot update or delete them directly. Do not retain the entire raw response, thumbnail URLs, or volatile statistics by default.

**Release Group**:
The master-level concept grouping editions of the same published release. It may be an album, EP, single, or another release type. A MusicBrainz Release Group and Discogs Master are external representations of this concept; a MusicBrainz Release or Discogs Release is one particular edition. Most UI shows only the title, not the words “Release Group,” “Master,” or “Album.” A Recording can link to two distinguished Release Groups: Original Release is its earliest credible official publication, while Primary Release is the most useful album context for display and browsing. They can be the same or different, and either stays absent when the evidence is ambiguous; see [ADR-0008](adr/0008-provider-neutral-music-entities-and-user-data.md).
_Avoid_: Master Album (not every Release Group is an album, and “master” also suggests audio mastering)

**Original Release**:
The Release Group containing the earliest credible official publication of that exact Recording, regardless of whether it is a single, EP, Album, or another type. This is provenance, not necessarily the title/artwork shown most prominently. Stored internally as the Recording's `original_release_group_id` relationship. Leave it absent rather than guessing when multiple candidates remain equally plausible.

**Primary Release**:
The Release Group chosen as the most useful album context for a Recording. An eligible candidate must actually contain the exact Recording and must not be a compilation or remix by default. Prefer an eligible Album, then EP; a trusted provider album hint can choose among eligible candidates but cannot make an unrelated title eligible. Fall back to Original Release when there is no better album context, and leave Primary absent when equally useful candidates remain unresolved. Primary Release drives the usual displayed release title and default artwork, while Original Release remains available as provenance. Stored internally as `primary_release_group_id`. For an ordinary album track, Original and Primary normally point to the same Release Group.

**User**:
A person using the app to track their own Songs and Recordings through `song_user_data` and `user_recording_data`, with private Artist notes in `artist_user_data`. Multi-user support is a design goal; canonical Song, Artist, Recording, and Release Group metadata is shared, while all user-data layers remain private. Each User has a Platform Preference.

**Platform Preference**:
A User's own ranked list of which platforms they care about (e.g. YouTube Music, YouTube, Amazon Music, Spotify, in that order for one User; Spotify, YouTube for another) — an order, not just a set. Controls which Platform Links are shown for a Recording (unranked platforms are normally hidden) and which is shown as the primary "Play on X" action (the highest-ranked platform that has a match). If none match, an attached playable YouTube Item is the explicit fallback exception when one exists, even if YouTube was not ranked. YouTube is common but not guaranteed in the provider-neutral model; a Recording created from Spotify may have no YouTube Item.

**Lead Sheet / Score**:
An image or PDF of sheet music a user attaches to a Song. Private by default, even if the underlying work is public domain. Can only become visible to other users if a Site Admin personally vets it and marks it public — never automatic, never self-service by the uploading user.
_Avoid_: Chart (unless quoting a genre convention)

**Site Admin**:
A trusted role (the app owner, or someone they explicitly trust), distinct from an ordinary User, who can vet and approve content — e.g. marking a Lead Sheet as verified public domain, or merging duplicate Songs — for visibility across users.
_Avoid_: Moderator, curator

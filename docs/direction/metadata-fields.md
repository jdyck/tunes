# Richer metadata fields

**Songs**: writing credits and year remain the only structured metadata beyond title. A writing credit targets an Artist and retains its `composer`, `lyricist`, or generic `writer` role; do not reduce the model to person-only composer/lyricist columns. Year means year written ([ADR-0007](../adr/0007-original-dates-and-albums.md)); MusicBrainz sourcing is best-effort and commonly null because writing relationships often lack dates. A "part of" text field (e.g. *Show Boat*) is planned — see [musicbrainz-matching.md](musicbrainz-matching.md).

**Recordings**: [ADR-0008](../adr/0008-provider-neutral-music-entities-and-user-data.md) defines Recording as canonical and provider-neutral. Shared fields live on `recordings`; private saved state lives on `user_recording_data`. A temporary database trigger mirrors private values into the legacy combined-row columns only until the newly modeled client is deployed and those compatibility columns can be contracted. `kind` (`released` / `video_capture`) was the first decided field. Do not add `user_upload` to that enum merely to represent a future self-hosted source: source and release status are separate dimensions, and the status term for an unreleased audio-only/home recording is still undecided. Shared `artist`, `album`, `year`, `duration`, `key`, and `tempo` remain plain columns (free text/simple types, no controlled vocabulary):

- `artist` and `duration` may initialize from selected YouTube evidence but stay editable; provider album identity remains on `youtube_items` rather than being copied into the canonical album hint. `key` and `tempo` are hand-entered. Personal `tags` live on `user_recording_data`.
- `year` means performance date, not release date ([ADR-0007](../adr/0007-original-dates-and-albums.md)); partial-precision/range-capable `recording_date_start` and `recording_date_end` text fields supersede it when the [musicbrainz-matching.md](musicbrainz-matching.md) work lands. YouTube release/album metadata is not valid performance-date evidence. Until true performance-date sourcing exists, leave it blank rather than relabeling a release year.
- `album` is currently an editable provider/title hint, not evidence that Original Release has been identified. The target has two relationships to shared, master-level Release Groups: `original_release_group_id` for the earliest credible official publication and `primary_release_group_id` for the most useful album context. The already-stored MusicBrainz Release MBID is only a representative edition for source/detail use. Primary Release supplies the normal title/artwork, falling back to Original Release; see [musicbrainz-matching.md](musicbrainz-matching.md).
- `artist` is a single free-text publication-credit snapshot, not a replacement for structured credits, because attribution is genuinely ambiguous (e.g. Barney Kessel playing guitar on a Billie Holiday recording). It stays for display even after structured Song and Recording credits converge on the broad shared `artists` identity described in [artist-browsing.md](artist-browsing.md).
- `notes`, `rating`, `sort_order`, and personal organization such as `tags` live in `user_recording_data`; canonical performance facts remain on Recording.
- A selected YouTube/YouTube Music result retains its normalized provider snapshot in `youtube_items` and associates that item with the Recording; this includes accumulated source provenance plus nullable YouTube Music artist/album identities that cannot reliably be recovered from video-ID lookup. See [streaming-platform-links.md](streaming-platform-links.md).
- The free-text `name` field is unaffected by any of this — it's still the recording's own title/descriptor, separate from `artist`.
- Metadata from external sources can contain HTML character entities (for example, `&amp;`). **Decided:** decode via a shared utility when values are loaded into the recording edit form. Cleaning a value marks the field dirty, so saving persists the decoded text and stored data self-heals as recordings are edited — no backfill migration. Also apply the same decode to newly ingested external data so new rows arrive clean. Do not scatter one-off replacements through individual components.

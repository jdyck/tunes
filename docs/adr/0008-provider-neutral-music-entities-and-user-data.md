# Music entities are provider-neutral; personal state is separate User data

Decided July 2026 while reconciling Standards' language with MusicBrainz, YouTube Music, and a future Discogs integration.

**Song, Artist, Recording, and Release Group are canonical Standards entities.** External catalog entities identify or describe them but do not define whether they can exist. A Recording can therefore exist without a MusicBrainz match, including an uncatalogued YouTube performance capture, and one Recording can have several playable provider items or links.

**Artist means any entity that can receive a musical credit**, following MusicBrainz's useful broad sense: a person, group, orchestra, choir, character, or other credited identity. Artist kind is nullable when unknown; missing type must not be coerced to MusicBrainz's positive `Other` type. A writer is not a special person entity; composer, lyricist, and writer are roles on a Song-to-Artist credit. A group credit is stored as the group and must not be discarded, coerced into a person, or expanded into individual members without evidence. The existing person-only `people` / `song_writers` model and the currently User-owned `artists` table are transitional and must be reconciled into one shared Artist identity when this migration is implemented.

**A User's private state on an Artist lives in `artist_user_data`.** Personal notes and tags do not belong on the shared Artist identity. The current `artists` table combines those private fields with the identity needed by Recording credits, so the private payload must move before that table is repurposed; one row per User and Artist is sufficient.

**Recording is the provider-neutral, Song-scoped identity for a specific recorded performance or version.** MusicBrainz Recording is an optional external identity on it, not the local entity itself. Provider items such as a YouTube video are sources or playable manifestations of the Recording; they are not mutually exclusive Recording subtypes. An official YouTube Music song can therefore belong to a Recording that also has a MusicBrainz Recording identity and links on other platforms, while an uncatalogued video capture remains a Recording with only its YouTube Item. When an external Recording or long provider item covers multiple Songs, the current model permits separate Song-scoped Recordings to reference that source; a richer multi-Work Recording model is out of scope until explicitly designed.

**A User's private state on a Recording lives in `user_recording_data`.** Saving a Recording, personal notes, rating, preferred order, and other genuinely personal fields do not make a second kind of Recording. One row per User and Recording is both the saved relationship and its private data; deleting it unsaves the Recording without deleting canonical metadata. This parallels the separation between canonical Song facts and `song_user_data`, even though the table names use the language chosen for each layer. The current `recordings` rows still combine canonical and User-specific fields; separating them is a future data migration, not a drive-by rename.

**Provider-specific metadata that cannot be reliably reconstructed is retained as a normalized provider snapshot.** For YouTube/YouTube Music, use one `youtube_items` concept globally identified by the shared YouTube video ID rather than separate `ym_tracks` and `ym_videos` tables. Retain discovery provenance that can include either search source, manual URL entry, or multiple paths. Normalize the search category to `song` when YouTube Music song or official Topic evidence exists and `video` otherwise; a manual URL starts as `video` until richer evidence exists, and this category remains independent of Recording Kind. Preserve useful stable fields, including nullable YouTube Music artist and album IDs/names and duration, because lookup by video ID does not reproduce all search-result metadata. Associate items to Recordings through `recording_youtube_items` so one Recording can have several items and one long item can support separate Song-scoped Recordings. Do not persist an entire raw provider response or volatile statistics merely because they were returned.

**Release Group is the master-level publication concept.** It can represent an album, EP, single, or another release-group type. MusicBrainz Release Group and Discogs Master are provider representations of this Standards concept; a MusicBrainz Release or Discogs Release is a particular edition beneath it. The app normally displays the Release Group's title without exposing “Release Group,” “Master,” or “Album” as UI chrome.

**A Recording can have both an Original Release and a Primary Release.** Both relationships point to Release Groups, not particular editions. Original Release is the earliest credible official publication of that exact Recording, even when it is a single. Primary Release is the most useful album context for display and browsing: consider only groups that contain the exact Recording, exclude compilations and remixes by default, prefer Album then EP, and fall back to the Original Release when no better album context exists. A trusted provider album hint may choose among eligible candidates; it does not make an unrelated title eligible. The two relationships normally point to the same Release Group for an album track and can differ when a single predates an album. Either remains null when the evidence is insufficient, and the matching flow preserves ambiguity rather than manufacturing a choice. Internal foreign keys use the unambiguous names `original_release_group_id` and `primary_release_group_id`.

## Consequences

- Song writing credits and structured Recording performer credits converge on the same shared `artists` identity; the credit role belongs on the relationship.
- `Recording.artist` can remain an editable credited-as snapshot for display even after structured Artist relationships exist.
- Future schema migrations introduce `user_recording_data`, reconcile `people` / `artists` (including private `artist_user_data`), and store normalized YouTube Items plus their Recording associations without requiring every Recording to match an external catalog; the handoff scope lives in [canonical-entity-migrations.md](../direction/canonical-entity-migrations.md).
- Release Group identity is first-class and provider-neutral; edition identity remains subordinate and optional.
- Recording detail and default artwork use Primary Release when present, then fall back to Original Release. Original Release remains available as provenance even when it is not the headline context.
- Provider IDs and snapshots support canonical entities. They do not replace them.
